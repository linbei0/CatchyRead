# 播放器面板紧凑化设计

**日期：** 2026-04-05  
**范围：** 内容页播放器浮层  
**目标：** 在不改变现有核心交互的前提下，让播放器面板更紧凑、更聚焦，减少纵向占用。

---

## 背景

当前播放器面板整体视觉质感较好，但纵向占用偏大。顶部品牌区、主信息卡、段落入口卡、主播放按钮与底部次级操作之间的留白都比较充裕，导致用户在较低视窗中更容易感到“撑”，主播放区也没有被进一步凸显。

用户希望“再压缩一点”，并明确选择了“适度压缩，同时弱化次要信息，让主标题和播放区更紧凑”的方向。

## 设计目标

1. 保持现有信息架构与操作路径不变。
2. 通过缩小间距、字号和按钮高度降低整体面板高度感。
3. 弱化次要信息卡与次级操作的视觉权重。
4. 保持主标题、模式切换和主播放按钮的识别度。
5. 不引入新的折叠逻辑、静默 fallback 或协议变更。

## 非目标

- 不修改后台消息协议或播放器状态流。
- 不改变“更多”面板与“全部段落”面板的交互语义。
- 不做新的响应式断点设计，仅在现有结构上收紧布局。

## 方案对比

### 方案 A：纯样式压缩

仅调整 `padding`、`gap`、按钮高度和字号。  
优点是风险最低，缺点是体感变化有限。

### 方案 B：样式压缩 + 信息层级微调（采用）

在方案 A 基础上，进一步压低顶部区、状态区、段落入口卡和次级按钮的存在感，让主标题与主控按钮获得更高聚焦。  
该方案不改变交互，却能带来更明显的“紧凑”观感。

### 方案 C：结构性折叠

把部分说明文案或底部控制折叠进“更多”。  
压缩幅度最大，但超出本次“轻量收紧”的范围，也会改变用户习惯。

## 采用方案

采用 **方案 B**。

### 顶部区

- 收紧面板整体 `gap` 与 `padding`
- 缩小品牌标题与胶囊标签尺寸
- 缩小顶部图标按钮尺寸

### 主信息区

- 收紧 hero 区块内边距
- 模式切换胶囊整体变薄
- 主标题字号下调一档，但保持为最强视觉元素
- 状态徽标与段落序号缩小并靠右收束

### 段落入口卡

- 保留“查看全部段落”的入口语义
- 收紧卡片 `padding` 与圆角
- 压缩文案字号与卡片内部行距，降低次要信息占比

### 进度区

- 缩小轨道上下间距与说明文字尺寸
- 保持进度条可点击区域和可访问性

### 控制区

- 上一段 / 播放 / 下一段按钮整体缩一档
- 底部四个次级按钮进一步降低高度和标签字号
- 保持主播放按钮依然是视觉焦点

## 实现影响

- 主要修改 `src/ui/content/player-view.ts` 中模板和 `buildPlayerViewCss`
- 通过 `tests/player-dom.test.ts` 补充紧凑布局断言，覆盖关键尺寸与文案结构

## 测试策略

1. 先在 DOM 测试中添加失败断言，描述预期的紧凑样式。
2. 修改模板和 CSS，使新增断言通过。
3. 执行：
   - `npm test`
   - `npm run typecheck`
   - `npm run build`

## 风险与控制

- **风险：** 压缩过度导致可读性下降  
  **控制：** 只压一档，不隐藏主信息，不改变交互层级

- **风险：** 测试只验证字符串，无法完全覆盖视觉  
  **控制：** 选择关键样式 token 做稳定断言，确保结构和主要尺寸可回归

## 第二轮微调（主控按钮与底部留白）

### 新反馈

用户在第一轮紧凑化后继续反馈：

1. 中部三颗主控按钮仍然偏大
2. 底部四个按钮下方仍有明显空白

### 第二轮目标

1. 再缩小主控按钮尺寸与间距
2. 同步缩小底部四个次级按钮
3. 继续压缩面板底部留白，让按钮区域更贴近面板底边

### 第二轮采用方案

继续采用 **方案 B**：

- 主控按钮与底部按钮一起缩小
- 面板底部 padding、分隔区 spacing 与主控区 gap 再压一档
- 不改按钮数量，不改交互逻辑，不把按钮折叠进更多面板

### 第二轮实现点

- 下调 `.panel` 的整体高度或底部空间占用
- 下调 `.transport` 的 `gap`、`.transport-button` 的高度与圆角
- 下调 `.secondary-controls` 的 `padding-top`、`gap`
- 下调 `.compact-action` 的高度、字号、内边距
- 保持主播放按钮仍为主视觉，但避免明显“过高”
